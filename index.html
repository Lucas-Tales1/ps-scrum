Scrum no Bem-te-ouvi

Adaptação do framework ágil para o contexto do PDS Corporativo.

Contexto do Projeto

No projeto Bem-te-ouvi, utilizamos o Scrum para gerenciar o desenvolvimento da nossa plataforma de streaming focada em músicos independentes. Dado o contexto acadêmico do próximo semestre, nosso foco muda da codificação pura para a gestão eficiente do ciclo de vida da aplicação.

As Sprints serão ajustadas para ciclos semanais, permitindo que a equipe valide rapidamente funcionalidades críticas, como o upload de álbuns e a reprodução de músicas, garantindo que o valor entregue esteja alinhado com a visão de "dar visibilidade a talentos pouco conhecidos".

Pilares Aplicados

Transparência

O Backlog do Produto (baseado nos Requisitos Funcionais F01-F11) fica visível para todos no GitHub Projects.

Inspeção

Revisão constante dos protótipos de alta fidelidade e do código Django para garantir a usabilidade (NF02).

Adaptação

Ajustes rápidos no escopo caso uma ferramenta (ex: player de áudio) apresente dificuldades técnicas.

Papéis e Responsabilidades

A equipe é formada por André Luiz, Marcos Alexandre, Lucas Tales e Manoel Pinto. Abaixo, definimos como as responsabilidades do Scrum são distribuídas no nosso contexto:

Product Owner (PO)

Focado na visão de negócio descrita no Documento de Visão. Decide o que é prioritário para o músico independente.

  • Priorizar a "F02 - Criar álbum" sobre a "F05 - Doar"
  • Validar se as telas seguem os protótipos
  • Gerenciar o Product Backlog no GitHub

Scrum Master

Responsável por garantir que o processo flua sem impedimentos técnicos ou de infraestrutura.

  • Resolver conflitos de merge no Git
  • Garantir o ambiente Docker configurado
  • Facilitar as Dailies no Discord/Meet

Developers

Toda a equipe atua no desenvolvimento full-stack (Django + Templates), implementando os Casos de Uso.

  • Implementar Views e Templates (ex: CDU-010)
  • Criar testes para garantir desempenho (NF03)
  • Realizar Code Review dos Pull Requests

Ciclo da Sprint

O fluxo de trabalho no Bem-te-ouvi é guiado pelos artefatos de documentação já produzidos. As cerimônias acontecem da seguinte forma na prática:

Figura 1: Fluxo Iterativo do Bem-te-ouvi

  • Sprint Planning 1

    Entrada: Requisitos Funcionais (Tabela F01-F11).

    O time seleciona itens de Alta Prioridade.
    Exemplo: Para a Sprint 1, selecionamos o CDU-001 (Ouvir música) e o CDU-011 (Registrar). A meta é permitir que um Visitante crie conta e ouça uma faixa demo.

  • Daily Scrum 2

    Foco: Sincronização técnica.

    Check-in rápido de 15 min.
    Exemplo: "Ontem finalizei o HTML do Login (CDU-010). Hoje vou integrar com a autenticação do Django. Estou travado no carregamento dos arquivos estáticos (CSS)."

  • Sprint Review 3

    Entrada: Funcionalidade "Pronta".

    Demonstração do software rodando.
    Exemplo: Mostrar ao professor o fluxo de um Músico cadastrando um evento (CDU-007) e validar se a interface está intuitiva conforme o Requisito Não-Funcional NF02.

  • Sprint Retrospective 4

    Foco: Melhoria do Processo.

    O time discute o que pode melhorar.
    Exemplo: "Precisamos melhorar as descrições dos Pull Requests" ou "A comunicação sobre mudanças no banco de dados precisa ser mais clara".

Práticas e Ferramentas

Adotamos práticas específicas para garantir a qualidade do código e a fidelidade à documentação do Bem-te-ouvi:

Modelagem por Casos de Uso

Em vez de User Stories genéricas, utilizamos CDUs detalhados (ex: diagrama em doc/cdu/cdu.md) para guiar o desenvolvimento e tratar fluxos de exceção.

Priorização MoSCoW

Baseada na tabela de Requisitos. Itens de prioridade Alta (F01, F02, F10) são desenvolvidos primeiro; itens de prioridade Média (F05, F07) ficam para o final.

Feature Branch Workflow

Cada Caso de Uso possui sua própria branch no Git. O código só é integrado à main após revisão e testes, mantendo o histórico organizado.

Definição de Pronto (DoD)

Um item só é concluído se: o código foi comitado, passou nos testes de desempenho (carregamento < 5s, NF03) e a interface segue o guia de estilos.

Validação com Protótipos

Antes de codificar, as telas são comparadas com os protótipos de alta fidelidade para garantir a experiência do usuário (UX).

Feedback Acadêmico

O ciclo de feedback inclui a avaliação do professor orientador, garantindo que o projeto atenda aos requisitos da disciplina de Gerência.
